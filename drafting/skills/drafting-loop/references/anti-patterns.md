# Anti-Patterns

| Pattern | Why it fails | Correct behavior |
|---|---|---|
| Build before Shape | Produces the wrong thing quickly. | Shape scope and criteria first. |
| Shape before reading materials | Invents requirements already answered by existing artifacts. | Inspect available materials first. |
| Confidence as evidence | Hides unverifiable claims. | Run checks or mark unknown. |
| Silent scope addition | Makes acceptance impossible. | Write a change note before work. |
| One-perspective review | Misses predictable blind spots. | Run phase review with Accuracy, Validity, Consistency where relevant. |
| Vague acceptance criteria | Cannot be accepted. | Convert to observable criteria. |
| Skipping challenge | First plan becomes locked-in. | In blueprinting, list rejected/deferred alternatives. |
| Monolithic increment | Hard to verify. | Split into independently checkable stages. |
| Documentation as delivery | Describes work instead of doing it. | Verify the actual deliverable. |
| Reinventing conventions | Conflicts with existing project. | Inspect existing patterns first. |
| Heroic cleanup | Expands beyond approved scope. | Finish the contract; record optional improvements. |
| Review as editing | Hides findings in silent edits. | Freeze artifact, review, then revise separately. |
| Rubric leakage | Produces answer-key-shaped output. | Use ground-truth isolation. |
| Process theater | Adds ceremony without evidence gain. | Process audit can choose no change. |
| Parent done before child done | Masks incomplete stages. | Parent status derives from child records. |
