# 非虚构技术系列文章创作工作流

> 从本项目（Promise Theory × Agentic AI 系列）的调研、审核、创作过程中提炼的通用工作流。适用于：基于某个理论框架或思想体系，创作系列技术/评论文章的场景。

---

## 总览

```
调研沉淀 → 对话探索 → 聚焦定位 → 初稿创作 → 全量审核 → 修复迭代 → 终审放行 → 逐篇发布
```

每个阶段有明确的产出物和检查点。不要求线性走完——可以回跳，但不能跳过。

---

## 一、调研沉淀

**做什么**：把原始材料（书籍、论文、报告、实践案例）消化成结构化的内部参考文档。不是直接写文章，是建立"知识底座"。

**具体动作**：
- 通读原始材料，提取核心概念、关键论点、争议点
- 按主题组织成参考文档（如本项目的 `taleb-incerto-complete-guide.md`、`antifragile-ai-agent.md`）
- 标注每个概念的来源、原著的准确表述、已知批评
- 内部参考文档可以长、可以全——它不是给读者看的，是给后续创作提供弹药库

**产出物**：内部参考文档（不发布）

**检查点**：
- [ ] 每个核心概念有原著出处的准确引用
- [ ] 概念的批评和争议被记录，不只记正面
- [ ] 内部参考和面向读者的文章明确分开

**本项目的教训**：
- 一开始写了一份 35000 字的 `incerto-ai-agent.md`，既是调研又是文章，太长、无渐进推导、读者会迷失。后来拆成"内部概念保留 + 精简系列文章"。**调研文档和读者文档要分开。**

---

## 二、对话探索

**做什么**：通过和用户的开放对话，探索理论框架和目标领域的交叉点，发现真问题和伪问题。

**具体动作**：
- 从一个开放话题起步（本项目是"不确定性"）
- 逐个深入子话题，用"反驳与回应"的方式压力测试论点
- 记录哪些论点站得住、哪些站不住、哪些需要限定条件
- 在对话中形成"定位共识"——这个理论框架在目标领域里到底是什么角色（不是银弹、不是定论，是某种视角/约束/补充）

**产出物**：对话中形成的论点清单和定位共识（记入工作日志）

**检查点**：
- [ ] 至少经过一轮"如果有人反驳 X，怎么回应"的压力测试
- [ ] 理论框架的定位被明确（设计基础？评价框架？约束式原理？）
- [ ] 理论本身争议性被讨论，不是回避

**本项目的教训**：
- 用户提了 10 个常见反驳（SOTA 升级论、个人体验论等），每个都暴露了论证的隐藏假设。如果跳过对话直接写文章，这些隐藏假设不会被发现自己发现。

---

## 三、聚焦定位

**做什么**：明确这组文章和已有内容（如本项目里 PT 系列已有 12 篇）的关系——是补充？是替代？是外篇？

**具体动作**：
- 如果有前置系列：读完全部前置内容，逐篇找交叉点
- 明确新文章的定位（本项目：T 三篇是 PT 12 篇之后的"外篇"，补 PT 没说透的尾部风险）
- 对每个交叉点决定：简短带过并标注"前置已讲"？展开前置没讲的视角？还是完全重复（要避免）？
- 确定篇幅、风格、文件结构

**产出物**：聚焦定位文档（可记入工作日志，不单独成文）

**检查点**：
- [ ] 新文章和已有内容的关系明确（互补/替代/外篇）
- [ ] 每个交叉点的处理策略确定
- [ ] 篇幅和风格有明确约定

---

## 四、初稿创作

**做什么**：按聚焦定位写初稿。

**具体动作**：
- 按 `writing-style-profile` 写（如果有的话）
- 每篇有一个核心论点，不堆砌
- 跨领域类比帮助理解，不是装饰
- 系列文章每篇开头/结尾明确和前后篇的衔接关系
- 术语仅在用到时简短解释，不单独铺陈理论

**产出物**：初稿文件

**检查点**：
- [ ] 每篇有明确的核心论点
- [ ] 系列文章前后篇衔接清晰
- [ ] 风格符合 writing-style-profile（如有）
- [ ] 术语解释不喧宾夺主

---

## 五、全量审核（三轮）

**做什么**：对全部文章做三轮系统性审核，每轮关注不同维度。

### 第一轮：事实与时事性审核

**检查什么**：
- 引用的概念是否和原著/原始来源准确一致
- 具体版本号、产品名、数据是否有事实风险
- 外部引用是否可核实（标注"待核实"的不算问题，伪装成已核实的算）
- 微信摘要等衍生内容是否和正文一致

**典型问题**：
- wechat-summaries 和正文脱节（本项目 P0 级问题）
- 具体版本号过时风险（如"GPT-5.3"）
- 概念使用和原著定义有偏差

### 第二轮：逻辑与论证审核

**检查什么**：
- **稻草人论证**：是否把"传统/对手能做的事"说成"做不到"，反过来证明自己的方案？这是最需要警惕的系统性问题。
- **自证陷阱**：是否把"自己设计的机制"混进"理论原本就有的"，让论证看起来比实际更强？
- **前后矛盾**：同一概念在不同篇目里的定义是否一致？对同一事物的评价是否矛盾？
- **论证-结论匹配**：每个子论证是否真的支撑最终结论？有没有子论证其实是弱论证但被当成强论证用？
- **隐藏假设**：每个论点是否藏着"在平均斯坦成立但在极端斯坦失效"的假设？

**典型问题**：
- PT03"传统能加评估"被说成"传统不能内建质量保证"（稻草人）
- PT04"信任分机制"被混进"Promise Theory 原本就有的"（自证陷阱）
- 对 Hermes Agent 一边说"最接近反脆弱"一边说"是火鸡问题再现"（前后矛盾）

### 第三轮：一致性与实践指导性终审

**检查什么**：
- **前后文一致性**：修改后是否引入新的衔接断裂？修改了 A 处，B 处引用 A 的内容是否还准确？
- **论证支撑**：修改后的论证是否还能支撑结论？降级后的论证是否诚实但足够？
- **实践指导性**：读者读完能不能动手？哪些篇直接指导工程实践？哪些只提供概念框架？缺的指导是否被坦诚承认？

**典型问题**：
- 修改后 PT05 段落顺序导致逻辑流断裂（修复引入的新问题）
- 修改后论证从"做不到"降级为"不默认做"，结论是否还成立（本项目：成立，靠三条强论证支撑）

**产出物**：审核报告（本项目产出了 `full-review.md`、`strawman-review.md`、`final-review.md`）

**检查点**：
- [ ] 三轮审核都执行了
- [ ] 稻草人论证被系统性排查（不只看单篇，看全部对比段落）
- [ ] 自证陷阱被排查（"理论给的"和"自己设计的"是否被诚实区分）
- [ ] 审核报告按严重度分级（P0必须修/P1强烈建议/P2建议/P3可选）
- [ ] 修改引入的新问题被检查（修改后重新读衔接处）

---

## 六、修复迭代

**做什么**：按审核报告修复，每处修复后检查是否引入新问题。

**具体动作**：
- 按严重度优先级修：P0→P1→P2→P3
- 每处修复后，检查引用该处内容的其他篇目是否需要同步调整
- 修复措辞时注意区分：
  - "传统做不到" → 应改为"传统不默认做"（诚实但弱）
  - "我们做了/实现了" → 应改为"可以考虑的设计方案"（未实现的东西不要说成已实现）
  - "理论给的概念" vs "自己设计的机制" → 显式区分
- 修复后做一次衔接检查：修改处的上下文逻辑流是否还顺

**产出物**：修复后的文件 + 工作日志记录

**检查点**：
- [ ] P0 全部修复
- [ ] 每处修复检查了引用链（改了 A，B 引用 A 的地方还准确吗）
- [ ] 修复引入的新衔接问题被处理
- [ ] "理论给的"和"自己设计的"被显式区分
- [ ] 未实现的机制被标注为"设计方案"而非"已实现"

---

## 七、终审放行

**做什么**：修复完成后做最终审核，确认可以进入发布流程。

**三个维度**：
1. **一致性**：全部文章前后文无矛盾。已发布的旧措辞和修改后的新措辞有轻微不一致可接受（理解为"精确化"），但不能是矛盾。
2. **论证支撑**：核心结论被子论证充分支撑。强论证至少 2-3 个（不可被"对手也能做"反驳的结构性差异），弱论证不能被当成强论证用。
3. **实践指导性**：至少有若干篇直接指导工程实践。缺的指导（如 benchmark 方案）被坦诚承认是开放问题，不假装解决。

**产出物**：终审报告（本项目 `final-review.md`）

**检查点**：
- [ ] 一致性通过
- [ ] 论证支撑通过（强论证≥2，弱论证不被当强论证用）
- [ ] 实践指导性通过（有可直接动手的篇目，缺的被坦诚承认）
- [ ] 剩余开放问题不构成发布阻碍

---

## 八、逐篇发布

**做什么**：按计划逐篇发布，每篇发布后记录进度。

**具体动作**：
- 发布前确认该篇的最终版本（发布后不可改的约束要提前想清楚）
- 发布后记录到工作日志（哪篇、哪天、反馈如何）
- 如果发布后发现新问题：已发布的不能改，未发布的可以同步调整；如果已发布篇目的问题影响后续篇目，在后续篇目里做隐性修正（不点名纠错，但把正确版本写进去）

**检查点**：
- [ ] 发布进度被记录
- [ ] 已发布篇目的问题在后续篇目里被隐性修正（如需要）

---

## 附：关键原则备忘

1. **调研文档和读者文档分开**——内部参考可以长和全，读者文档要精简有渐进。
2. **理论给的和自己设计的要显式区分**——不要把自己的工程设计混进理论框架里，让论证看起来比实际更强。
3. **稻草人论证是系统性风险**——"传统做不到"几乎总是错的，正确的说法是"传统不默认做"。
4. **修改会引入新问题**——每处修复后检查衔接，不能改了 A 忘了 B。
5. **未实现的要说成设计方案**——不要用"我们做了"的口吻讲还没实现的东西。
6. **审核要分轮次**——事实/逻辑/一致性三轮各有侧重，一轮审不全。
7. **对话探索不能跳过**——用户提的反驳会暴露你自己看不到的隐藏假设。
