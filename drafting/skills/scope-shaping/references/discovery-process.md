# Discovery Process

Use this reference to preserve delivery-grade requirement discovery while staying universal across work kinds.

## Material scan

Before asking questions or writing scope, inspect available materials:

- existing tracks, scope notes, requirements, briefs, plans, delivery records;
- knowledge bases, logs, source artifacts, documents, data, tests, previous outputs;
- current user request and any supplied examples.

Stop scanning when more context is unlikely to change the next scope decision.

## Intent shaping

Extract:

- elevator pitch or one-sentence outcome;
- user/persona/audience;
- scenario or journey;
- real need and current pain;
- why now;
- constraints and non-goals;
- success criteria;
- risks and rollback needs.

## Challenge pass

Challenge before finalizing scope:

- vague words: better, cleaner, robust, simple, polished;
- overloaded terms with multiple meanings;
- hidden assumptions;
- over-engineering;
- missing non-goals;
- acceptance criteria that are not observable;
- broad scope that should split.

## Question rule

Ask one focused question at a time only when context cannot answer it. Prefer a recommended option with trade-offs when choices matter.

## Approach options

When scope choices matter, present 2-3 approaches:

```markdown
| Option | Scope | Pros | Risks | Verification | Recommendation |
|---|---|---|---|---|---|
```

## Output scale

| Work size | Output |
|---|---|
| Simple | compact standalone scope note |
| Normal | track folder with scope/requirements artifact |
| Large | parent track with scope-map and child stage tracks |
| Multi-deliverable | parent track with one child per deliverable or dependency stage |

## Required shaped-work content

- Goal / problem
- Persona, audience, or scenario
- Scope and non-goals
- Requirements or brief
- Acceptance criteria
- Verification plan
- Contract/data model if behavior crosses a boundary
- Risks / rollback
- Open questions
