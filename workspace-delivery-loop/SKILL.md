# Workspace Development Loop

适用于多模块项目的中大型交付。强调"需求↔计划↔验证↔代码"可追溯，默认 TypeScript。

## 适用 / 不适用

- 适用：跨模块改动、需要验收口径或版本发布
- 不适用：10 行内小修复、纯格式化

## 写作规范

- 禁用词：可能/大概/尽量/差不多/视情况 → 改为：必须/不允许/当…则…
- 验收口径必须可二元判定（如"当输入为空时，按钮禁用"）
- 范围必须明确（如"不涉及历史数据迁移"）

## PRD（中大型需求必须）

PRD 是"设计图"，vN 计划是"施工计划"。PRD 提供稳定口径，vN 计划基于 PRD 分解执行。

PRD 必备段落：
1. **背景与问题**：痛点 + 为什么现在做 + 已有约束
2. **目标**：可验收的语言（G1/G2/G3）
3. **非目标**：不做什么（防返工的关键）
4. **术语**：关键名词统一命名，避免换名
5. **场景**：至少 3 个关键场景（用户动作 → 系统结果）
6. **需求清单**：每条需求有 Req ID + 验收口径 + 验证方式

| Req ID | 需求 | 验收口径 | 验证方式 | 优先级 |
|---|---|---|---|---|
| REQ-001 | ... | 可二元判定 | 命令/测试 | P0/P1/P2 |

7. **约束**：性能/安全/平台限制
8. **追溯矩阵**（实施侧维护）：Req ID → vN 计划 → 测试 → 代码路径

详细模板见 `references/prd-template.md`。

## 流程（10 步）

0) 明确需求归属模块
1) 写 PRD（中大型需求必须）→ `docs/track/PRD-<topic>.md`
2) 写 `docs/track/vN-<topic>.md`（基于 PRD 分解执行计划）
3) 建立验收标准与回归清单
4) 通过 **DoD Gate**
5) 通过 **Doc QA Gate**
5) Schema-first：先改 schema/types
6) 实现（分层：shared → background → UI）
7) E2E Gate
8) 回顾差异（写入 `vN-index.md`）
9) 交付与文档同步

## DoD Gate（强制）

通过以下条件才允许进入实现：
- 需求范围可判定（包含 Non-Goals）
- 验收标准可二元判断
- 数据模型已定义（schema-first）
- 依赖与影响模块明确
- **架构落点已确认**：每个模块的目标包/目录已写入计划文档
- 有最小验证路径
- **反作弊条款**：每条 DoD 必须包含"看起来通过但实际没做"的检测手段（如"所有旧类型引用已删除"）

## Doc QA Gate（强制）

检查 `vN-<topic>.md`：
- 术语一致，无"同义不同名"
- 关键约束清楚（不接受/不做）
- 验收标准覆盖核心路径与边界路径

## 执行循环

1) Red：先写最小验证或断言
2) **实现前重读**：重读计划的架构落点，确认代码在正确包/目录
3) Green：实现最小闭环
4) Refactor：清理结构
5) **架构自检**：确认代码在正确位置。若偏差，写 ECN 修正，**不允许用 compat shim 绕过**
6) 同步更新文档

## Phase 结束检查点（多 phase 强制）

每个 phase 结束，标记 done 前：
- [ ] 所有模块在计划指定的包/目录中
- [ ] 依赖链符合计划
- [ ] 无 compat shim 或 deprecated alias
- [ ] 下一 phase 的依赖已就绪
- [ ] 测试通过 + 构建成功
- [ ] AGENTS.md 已更新

## 遇到意外复杂度（必须停下）

触发条件（任一即暂停）：
- 需修改文件数 > 计划预估 2 倍
- 发现计划未覆盖的依赖/耦合
- 旧代码复杂度超出预期（500+ 行需重写）
- 编译错误需修改计划外模块

暂停后：停止写代码 → 评估偏差 → 必要时写 ECN → 更新计划。**不允许"先凑合以后再改"**。

## 模块抽取 / 迁移

1) 识别边界（输入/输出/依赖）
2) 确定落点（小范围 → shared，跨模块 → 新包）
3) API 设计（稳定接口 + 隐藏实现）
4) **迁移策略**：删除旧代码 → 逐个重写调用方（不允许双写/兼容层）
5) 验证 + 文档同步

**大规模迁移**（调用方 >5 个文件）：每次改 1-3 个文件的 import，每步都能编译。所有调用方迁移完毕后删除旧代码。不允许旧位置留 re-export。

## Change Control（ECN）

设计偏差时必须写 `docs/track/ECN-NNNN-<topic>.md`：原设计 → 问题 → 新设计 → 影响评估。

## Schema-first

新需求先改 schema/types，存储/UI 只能建立在最新 schema 之上。

## AGENTS.md 同步

目录职责变化必须更新同级 AGENTS.md。发布前：`rg --files -g 'AGENTS.md'` 检查。

## 交付清单

- 代码实现完成
- **架构对齐**：模块在正确包/目录，无 compat shim
- 验收通过（测试或手动验证）
- CHANGELOG + AGENTS.md 同步

## 提交规范

- 一条提交只做一件事
- 格式：`feat(scope): 描述` / `fix(scope): 描述` / `docs(track): 描述`

## 压力场景

| 说法 | 正确做法 |
|---|---|
| "时间不够" | 至少保留 DoD + Schema-first + 验收记录 |
| "需求不清" | 暂停实现，先补齐 vN 文档 |
| "旧代码编译不过" | 不允许加 shim，必须重写调用方 |
| "文件太大不知道怎么删" | 拆小步骤，每次迁移 1-3 个调用方 |
| "测试通过了但架构不对" | 测试通过 ≠ done，必须检查代码位置+依赖链 |
| "先放这里以后再搬" | 实现前确认架构落点，写入计划 |

## Red Flags（必须停下）

- 需求没有验收标准
- 未更新 schema 就改 UI/逻辑
- 口头变更但无 ECN
- **用 compat shim 延续旧架构**（compat type / fallback function / deprecated alias）
- **代码不在计划指定的包/目录中**

## 常见反模式

| 反模式 | 正确做法 |
|---|---|
| 未更新 schema 就改 UI | Schema-first |
| 需求只口头确认 | 写入 vN 计划 |
| 实现与验收标准脱节 | DoD 绑定验证命令 |
| 用例没有边界条件 | 对照清单逐条检查 |
| 用 compat shim 掩盖偏差 | 删除旧代码，重写调用方 |
| 测试通过就标 done | done 前检查：代码位置+依赖链+无兼容层 |
| 遇到复杂度继续推 | 满足暂停条件时停下，评估后更新计划 |
| 计划写了但不对照实现 | 每次实现前重读计划 |

## 追溯矩阵

每条验收标准 → 一个实现点（文件）+ 一个验证点（测试）。断链必须补齐。
