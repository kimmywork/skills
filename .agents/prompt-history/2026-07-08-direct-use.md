我理解这不是dual-mode。无论是否有产出物还是handoff，对于用户来说都是友好的。哪怕是handoff，也可以直接提醒用户下一个技能可以怎么触发。所以任意入口开始这个就够了。

如果可能有冲突：比如 quality-reivew、audit-trail、和acceptance-gate，那这几个之间相互判断然后告知用户是否选择 x，而不是 y？这样交叉的情况最小，比每个skill都引用一下intent routing要简单得多。

---

1. 不要强行给每个SKILL统一格式。
2. 充分利用 SKILL.md frontmatter 的 description 字段，那里会更好呈现入口协议。
3. 不需要每个SKILL都有最临近，不要把用户当成傻子，plan-execution 谁都知道是执行，别的也没有执行的目的。
4. 没必要指导 SKILL 的话术。
5. next phase或者下一个推荐技能不要使用技能名字或者SKILL.md引用，而是使用soft reference，阶段名或者任务类型 —— 这样保持关联的情况下给足用户选择空间。
6. 不要强调输出风格，输出的内容我们有模板，非模板化和标准化的内容没必要指定格式。

以上的原则重点其实都是：不要把多种同样的、重复的、模板化的内容批量应用到所有的SKILL中，那样最终也会因为分散的修改变得不一致。每个 SKILL 都有他特殊的适用情况、独立性和特点，产出物和能够做到的效果也不同，所以不需要
这种形式的统一。
